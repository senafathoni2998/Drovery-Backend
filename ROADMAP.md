# Drovery — Feature Recommendations

Prioritized advice on **what to build next**, grounded in what exists today. Ordered by
impact-to-effort. ✅ = already added in the latest round of work.

## ✅ Just shipped (this round)
- **✅ Interactive API docs (OpenAPI / Swagger)** — browsable docs at **`/api/v1/docs`** (+ machine-readable `/api/v1/docs-json`) over the full ~79-route surface. `@nestjs/swagger` + the CLI plugin (`nest-cli.json`) auto-infers request/DTO schemas from the TS types + JSDoc at build time (no hand-decorating 100+ fields). Security schemes: **Bearer** (global default; `Authorize` once and Try-it-out sends it) + an **`x-ingest-key`** API-key scheme on the `@Public` drone `/ingest/*` routes. The runtime `{success,data,timestamp}` envelope is made **first-class in the spec** — a post-process rewrites every documented 2xx to `allOf[ApiEnvelopeDto,{data}]` and injects `400/401/500 → ApiErrorDto`, so a generated client deserializes the real shape. Public routes are correctly lock-free via a `@PublicApi()` composite (`@Public` + `ApiSecurity({})`) so the guard opt-out and the doc opt-out can't drift. `SWAGGER_ENABLED` kill-switch (on by default — it's the portfolio showcase). Designed grounded in the code + adversarially reviewed via **Opus** workflows (9 findings fixed: the doubled-prefix Try-it-out break, public-routes-falsely-authed, the envelope/error/orphaned-models cluster, flagship `AuthTokensDto`, prod-exposure doc, env doc; 2 refuted — confirmed **no raw-entity / secret-field leakage**). Verified hardware-free against the BUILT doc (the plugin only runs at `nest build`, not ts-node): 79 paths, both schemes, envelope+error wrapping, no leaks. *(Deferred: curated `*ResponseDto` for the ~71 non-auth endpoints whose response bodies are still untyped-object.)*
- **✅ Bidirectional drone commands (operator → drone control)** — telemetry was one-way (drone → backend); there was no way to command an in-flight LIVE drone (the `beginReturnToBase`/`completeReturnToBase` transitions existed but were exposed **nowhere**). Now an ADMIN issues a command on a new **`DroneCommand` outbox** table (`POST /admin/deliveries/:id/commands`, `Role.ADMIN`); the (mock) drone **polls + acks** it over the **same `/ingest` transport** telemetry uses (`GET /ingest/commands`, `POST /ingest/commands/:id/ack`, `DroneAuthGuard`); and the **ack is the sole trigger** that drives the delivery, reusing the existing single-winner CAS transitions (`RETURN_TO_BASE → beginReturnToBase → RETURNING`; `ABORT → failExceptional → DELIVERY_FAILED`). Issuing/polling never touch the delivery — the backend never claims the drone is returning until it actually accepts. v1 ships exactly those two types (others — HOLD/LOITER/DIVERT — need a new lifecycle state/route model, deferred). Correctness: a **partial-unique index** allows at most one open command per delivery (concurrent issue → DB-level 409); the ack **claim CAS** makes a duplicate/replayed ack a 409 with no double-transition; an ack racing the drone's own telemetry phase or the watchdog collapses safely (idempotent CAS, `appliedTransition` records who won); an operator-stamped reason (e.g. a non-refunding `RECIPIENT_UNAVAILABLE`) is preserved. **Self-heal**: TTL expiry (5 min) folds into the watchdog tick (lazy poll/ack guard + sweep), and a command stranded ACKED-but-untransitioned (crash between claim and transition) is **reconciled** by the watchdog re-driving the operator's original type+reason. **Security**: the ingest HMAC now binds `method`+`path` (not just the body), so a captured signature can't be retargeted across routes/drones/commands. **Observability**: `drovery_drone_commands_total{type,result}` (issued/fetched/acked/superseded/rejected/expired) + a result-labelled time-to-ack histogram; owner notifications/WS come free via the reused transitions. Designed + adversarially reviewed via **Opus** workflows (10 findings fixed: stranded-ack recovery + the HMAC binding + 8 metric/correctness items; 6 refuted); +32 tests. Live-verified hardware-free against real PG/Redis (20 checks: full issue→poll→ack→RETURNING, ABORT→FAILED, partial-unique 409, per-droneId scoping, duplicate-ack 409, SIMULATED 422, telemetry race, expiry, and stranded-ack reconciliation).
- **✅ Stuck-delivery watchdog (self-healing reaper)** — closes two reliability gaps left by #15/#16: a LIVE drone that loses comms mid-flight, or a return flight that dies, would otherwise sit in-flight **forever**. A worker-tier **Redis-coordinated repeatable scan** (`upsertJobScheduler`, mirrors the recurring materializer) reaps `LIVE` in-motion deliveries (`DRONE_ASSIGNED`/`PICKUP_IN_PROGRESS`/`IN_TRANSIT`/`RETURNING` — **never** the no-time-bound `AWAITING_HANDOFF`) whose **telemetry has gone silent** past `WATCHDOG_SILENCE_MS` (default 10m ≫ the ~5s position cadence), transitioning them to `DELIVERY_FAILED` + customer refund by **reusing** `failExceptional`'s single-winner CAS (idempotent, multi-replica safe; a real frame that landed first no-ops the reap). Silence is keyed on the **tracking row's** `updatedAt` (bumped by every position frame), *not* the delivery's phase-change time — both the SQL gate and the ordering use it, so a healthy long-haul flight can never crowd a genuinely-silent one out of the bounded batch (a starvation bug the **Opus** review caught). `SIMULATED` deliveries are out of scope (the sim drives them). Reason already stamped at an abort (e.g. `WEATHER_ABORT`) is preserved, not overwritten. Kill-switch (`WATCHDOG_ENABLED`) that tears down the persisted scheduler on a disabled boot; Prometheus `drovery_watchdog_reaped_total` + a last-scan **heartbeat gauge** (alertable) + the watchdog/recurring queues added to the queue-depth gauge. Designed + adversarially reviewed via **Opus** workflows (8 findings fixed: the starvation high + 7 lower); +9 tests. Live-verified hardware-free against real PG/Redis (silent IN_TRANSIT/RETURNING reaped incl. the null-tracking + reason-preserve paths; fresh-telemetry, SIMULATED, and AWAITING_HANDOFF correctly untouched); live-verify caught a real DI wiring miss (the queue-depth gauge's new queues weren't registered in `MetricsModule`).
- **Live drone tracking** — backend simulates a moving drone; mobile polls every 4s and the marker **glides** between positions with a LIVE badge.
- **Status-change notifications** — in-app + local banner on each change, and **remote Expo push** to registered devices.

Email send is behind `MailService` (logs the link in dev; swap in SendGrid/SES). Mobile has Forgot/Reset Password screens + a wired "Forgot password?" link, with deep-link token prefill.

- **✅ Proof of delivery (P0)** — on `DELIVERED` the simulation auto-records a `ProofOfDelivery` (photo + final GPS + recipient + timestamp); `POST/GET /deliveries/:id/proof` lets the owner submit/view one. Image storage is behind `StorageService` (inline data URL / placeholder in dev; S3/Cloudinary when configured). Mobile shows a proof card on the delivery detail **and can capture a real photo** (expo-camera + current GPS → upload, replacing the placeholder).
- **✅ Security hardening** — refresh-token rotation + revocation (hashed store, `jti`, reuse → 401, logout endpoint), per-IP rate limiting (`@nestjs/throttler`, tighter on `/auth`), weak-JWT-secret boot guard in production, owner-scoped tracking lookups, and a CORS allowlist (`CORS_ORIGINS`). All verified live.

Public, un-throttled, k8s-ready. **Sentry error tracking** (`@sentry/node`, DSN-gated no-op) reports unhandled 5xx from the global exception filter in both the API and worker. **Prometheus `GET /api/v1/metrics`** (`prom-client`): default Node metrics, an HTTP histogram labelled by route *template* (cardinality-safe), and a `drovery_queue_jobs{queue,state}` gauge from BullMQ — the worker also serves it at `:9091/metrics`.

Validated with `kustomize build` + `kubeconform`; CI dry-runs on `kind`. *(Next: a real cluster run for actual scale-up numbers + Grafana dashboards.)*
- **✅ Real-time tracking (push, horizontally scalable)** — WebSocket tracking over raw `ws` (`WsAdapter`) backed by **Redis pub/sub**: the worker publishes each position/status change to `delivery:<id>:update`, and every API replica's subscriber fans it out to its local clients — so a worker-computed update reaches a client on **any** instance (the in-memory gateway was dead in the worker-split topology). **Auth + ownership** on every subscription (JWT in the `?token=` handshake, `findOne` ownership re-check; tokenless → `1008`). Polling (`GET /deliveries/:id`) is kept as an additive backstop — **mobile still polls; WS migration is a separate mobile-repo task.** `drovery_ws_connections` gauge added. Verified cross-process. *(Next: dedicated realtime tier so sockets scale apart from the API.)*
- **✅ Recipient handoff OTP (P1 #5)** — the drone now stops at a new **`AWAITING_HANDOFF`** state on arrival (the sim never auto-delivers); finalizing as `DELIVERED` + recording proof requires the recipient's **6-digit code** via `POST /deliveries/:id/confirm-handoff`. Code is SHA-256-hashed (plaintext returned once on create, never re-exposed — `handoffCodeHash`/`handoffAttempts` omitted from all reads), constant-time compared, single-use (atomic CAS), with an atomic **5-attempt lockout** (`423`). Owner-scoped; `401/409/400/404` on the error paths. Verified live + adversarially reviewed.
- **✅ Serviceability gate (P1 #8)** — "can we fly this?" checks: **service area** (point-in-circle, Greater Jakarta + Bandung), **no-fly zones** (route-vs-circle geometry, Jakarta airports), and **weather** (real-or-mock `WeatherService`, OpenWeather when keyed, **fail-open**). `POST /pricing/estimate` returns an advisory `serviceability` block; `POST /deliveries` rejects unflyable routes (**422** out-of-area / no-fly / unresolved-location, **503** weather hold) before any charge/queue. Reused `haversineKm` (moved to `common/geo-distance` to break a Pricing↔Serviceability cycle). Verified live + adversarially reviewed.
- **✅ Saved addresses / address book (P1 #7)** — `/addresses` CRUD (jwt, owner-scoped, default-first, atomic single-default via `$transaction`, per-user cap) with **geocode-on-save** (best-effort; stores coords so future deliveries skip the rate-limited geocoder) + `GET /addresses/recent` derived (deduped) from delivery history. lat/lng bounded to WGS84. Verified live + adversarially reviewed.
- **✅ Ratings & feedback (P1 #6)** — once a delivery is `DELIVERED`, the owner can rate it 1–5 stars + optional comment via `POST /deliveries/:id/rating` (upsert, owner-scoped, `DELIVERED`-only → `409`). Embedded in the delivery detail includes. Verified + adversarially reviewed.
- **✅ Notification preferences (P1 #9, part 1)** — `GET/PATCH /notifications/preferences` for per-user `pushEnabled` / `deliveryUpdates` / `promotions` / `quietHoursStart`/`End`. The in-app feed is **always** recorded; preferences gate only the **push** fan-out (`create()` takes a `category` and checks the matching toggle + quiet hours). Quiet hours are `[start,end)` wall-clock hours (wrap-around handled), evaluated in `NOTIFICATIONS_TZ` (**default `Asia/Jakarta`, not the UTC container**) and require both bounds together (`400` otherwise). Verified live + adversarially reviewed (the review caught the UTC-vs-WIB quiet-hours bug + the silent half-window).
- **✅ Admin / agent API (P3 #14, API)** — a coarse `Role` (USER/AGENT/ADMIN) + a global `RolesGuard` (inert without `@Roles`; role resolved fresh from the DB so promote/demote is instant, deny-by-default) gating an `/admin` surface in this repo. **Support agent inbox** (AGENT+ADMIN): list/view all tickets + **reply as AGENT** (persisted + attributed + fanned out live to the user via the existing tier-agnostic chat publisher — completes #9) + status transitions. **Delivery oversight** (ADMIN): cross-user list/detail, **force-cancel** (conditional CAS bypassing the gate + best-effort cleanup), **refund** (wallet credit — no Stripe refund API — idempotent, marks Payment REFUNDED). **Promo CRUD** (the deferred #11 admin piece). **Overview** stats. **User role management** (last-admin guarded). - **✅ Reorder + favorites (P2 #13)** — `POST /deliveries/:id/reorder` ("send again") clones a past delivery and reuses `create()` (fresh serviceability/pricing/payment, new tracking/handoff, no promo/credits carried over; immediate by default, optional pickup override). `Favorite` snapshots a delivery as a reusable template: `POST /favorites` (from a delivery id, owner-scoped), `GET /favorites`, `DELETE /favorites/:id`, `POST /favorites/:id/order`. Opus-reviewed (clean; one test-only fix); +9 tests. Live-verified. **P2 #10–#13 all complete.**
- **✅ Referrals + wallet credits (P2 #12)** — "invite a friend, both get credit." Each user has a unique `referralCode`; signup links an inviter's code (best-effort, never blocks signup) into a PENDING `Referral`; the referee's **first delivery** credits **both** parties $5 **exactly-once** (a `PENDING→REWARDED` single-winner CAS inside the create transaction). Wallet = denormalized `User.creditBalance` (DB `CHECK >= 0`) + an append-only `WalletTransaction` ledger; **spend at checkout** via `useCredits`, stacked after promo, debited atomically (conditional-decrement CAS so the balance never goes negative), Stripe charged `total − promo − credits` (free → skipped); **cancel refunds** spent credits (idempotent), reward not clawed back. `GET /wallet` (balance + ledger), `GET /referrals` (code + stats). Designed + reviewed via **Opus** workflows (review clean); +17 tests. Live-verified the full earn→spend→refund loop. *(P2 #11+#12 promo/credits complete.)*
- **✅ Promo codes at checkout (P2 #11)** — discount codes applied to the price: `PromoCode` (PERCENT/FIXED + min-order + max-discount cap + window + global cap + per-user limit) and a `PromoRedemption` ledger. `POST /promo/validate` previews (advisory, never throws); `POST /deliveries` takes an optional `promoCode`, **validated + redeemed atomically** with the delivery in one interactive transaction — global cap via conditional-increment CAS (throws→rollback on the last-slot race), per-user limit via a **partial unique index** (`WHERE status='REDEEMED'`) — so a code can't be over-redeemed across replicas and the Stripe charge == the discounted total (free order → Stripe skipped). Cancel **releases** the slot (REDEEMED→RELEASED + decrement, idempotent) so re-use works. Demo codes `WELCOME10`/`DRONE5` seeded via migration + `seed.ts`. Designed + reviewed via **Opus** workflows; +22 tests. Live-verified the discount, the 409 per-user/global guards, cancel-release→re-apply, and the ledger. *(wallet/credits → #12.)*
- **✅ Recurring deliveries (P2 #10, part 2)** — `/recurring-deliveries` CRUD (DAILY / WEEKLY-by-day, owner-scoped, pause/resume) + a **materialization worker** that auto-creates a delivery per occurrence by reusing `DeliveriesService.create()` (so each instance flows through serviceability/pricing/payment/handoff and the SCHEDULED+kickoff path). A **Redis-coordinated repeatable scan** (`upsertJobScheduler`, every 60s) claims each occurrence with an **atomic `nextRunAt` CAS** (multi-replica safe), materializes within a 6h lookahead, **skips missed occurrences** without backfilling, and is **at-most-once** (CAS-before-create — `create()` mints a trackingId + Stripe intent). Next-occurrence math is service-tz-correct (noon-anchored weekday; WIB-day-inclusive start/end bounds). Designed + reviewed via **Opus** workflows; +35 tests. Live-verified incl. the scan materializing a SCHEDULED delivery + advancing the cursor; review caught the start/end boundary + `?active` coercion bugs (fixed). *(P2 #10 complete.)*
- **✅ Scheduled deliveries (P2 #10, part 1)** — a future pickup window now **defers the whole lifecycle** instead of flying immediately. `POST /deliveries` derives the pickup UTC instant from `pickupDate + pickupTime` in the service tz (dependency-free `Intl` offset math), and a lead time > 60s (≤ 60-day horizon, else `400`) sets a new **`SCHEDULED`** status + `scheduledFor` and enqueues one delayed BullMQ **kickoff** job; at the window it **atomically flips `SCHEDULED → PENDING`** (enqueue-before-flip so a transient retry recovers) and starts the normal simulation. Immediate/past pickups behave exactly as before; cancel works while scheduled (removes the kickoff job); new `?status=scheduled` filter (kept distinct from in-flight `current`). Designed via a 2-lens panel workflow; +20 tests (339 pass). Live-verified incl. the kickoff **firing at the pickup instant** (`SCHEDULED → PENDING → CONFIRMED`); live-verify caught a real bug (BullMQ `queue.add` rejects `:` in a custom jobId) and the review caught a retry-stranding ordering bug — both fixed. *(Next: truly recurring deliveries.)*
- **✅ Realtime support chat (P1 #9, part 2)** — a SECOND WebSocket gateway at a distinct path **`/ws/support`** (the WsAdapter routes upgrades by exact pathname, so the mobile's tracking socket at `/` is untouched — verified live). Mirrors the tracking tier: JWT-handshake auth (`1008` if tokenless), per-ticket **ownership** re-checked at subscribe AND send, dedicated-connection **Redis pub/sub** fanout (`support:ticket:<id>:messages`) so a message accepted on any replica reaches every subscribed client on every replica. Messages persist in a new **`SupportChatMessage`** thread (idempotent backfill seeds existing tickets); a **SYSTEM auto-acknowledgement** is seeded on ticket creation. **REST backstop**: `GET/POST /support/tickets/:id/messages` (history + send, identical effect to the WS frame). Built **human-agent-ready** (USER/AGENT/SYSTEM roles + tier-agnostic publisher) — a future agent surface needs no schema change. `drovery_ws_support_connections` gauge. Designed via a 2-lens panel workflow; verified live (two clients realtime over the full publish→Redis→subscribe loop, both gateways coexisting) + adversarially reviewed (caught a non-atomic seed write + a migration UUID portability gap, both fixed). *(Next: the agent/admin surface — inbox, status transitions, optional AI-assist.)*

A coarse `Role` (USER/AGENT/ADMIN) + a `RolesGuard` (global, inert without `@Roles`, role resolved fresh from the DB → instant promote/demote) gate an `/admin` surface: **support agent inbox** (list/view tickets, reply as AGENT → live to the user via the existing chat publisher, status transitions — completes #9), **delivery oversight** (cross-user list/detail, force-cancel, refund-as-wallet-credit), **promo CRUD** (the deferred #11 admin piece), **dashboard overview** (counts/revenue), and **user role management** (last-admin-guarded). First admin seeded (`admin@drovery.com`).
15. ~~**Live drone telemetry ingestion** to replace the simulation (MQTT/drone gateway → worker tier → same tracking contract).~~ ✅ **Done.** A delivery is `SIMULATED` (default) or `LIVE` — a `TrackingSource` discriminator fixed at `create()` (`POST /deliveries` `trackingSource: 'LIVE'`), so a real drone and the sim are **interchangeable producers of the same tracking contract** and never both drive one delivery (LIVE enqueues **no** sim jobs; telemetry **refuses** a SIMULATED delivery). A transport-agnostic `TelemetryService.ingest()` validates a frame and applies it by **reusing** the existing primitives — the monotonic forward-only status CAS (`statusesBefore`), `TrackingService.updateTracking`, `TrackingPublisher` — so an out-of-order/duplicate/stale frame is a no-op, a terminal delivery can't be resurrected, and the live path **never auto-delivers** (a drone reports a *phase*, mapped to a status with **no** DELIVERED target — it stops at `AWAITING_HANDOFF`, finalized only by the recipient OTP). Primary transport = a hardware-free `@Public` `POST /ingest/telemetry` authenticated by a `DroneAuthGuard` (shared `INGEST_API_KEY`, constant-time; optional **timestamped HMAC** over the raw body with a freshness window — anti-replay; **fail-closed** if unset) — never reachable by a user JWT. Each LIVE delivery is bound to one `assignedDroneId` (a stranger gateway can't drive it). An optional `MQTT_URL`-gated subscriber is a documented no-op shell (the real-broker path, deferred). Verified end-to-end with a mock publisher (no hardware).
16. ~~**Delivery exceptions** (failed drop, weather abort, return-to-base) as first-class statuses with user comms.~~ ✅ **Done.** Three new statuses — `RETURNING` (transient: the drone flies the package home, marker keeps moving), `DELIVERY_FAILED` and `RETURNED_TO_BASE` (terminal) — plus a `DeliveryFailureReason` (`WEATHER_ABORT`/`UNSAFE_DROP_ZONE`/`MECHANICAL`/`RECIPIENT_UNAVAILABLE`/`ADMIN_ABORT`/`OTHER`) so the outcome is explainable rather than a `CANCELED` catch-all. The exception statuses live **outside** `STATUS_ORDER` (a tripwire test enforces it), so the linear forward CAS can never enter them or resurrect a terminal; each transition is its own **dedicated conditional CAS** (mirroring `adminForceCancel`) — single-winner, idempotent, cleanup+comms run exactly once. Fault-based refund: a drone/service-fault (`WEATHER_ABORT`/`MECHANICAL`/…) stops the sim + releases the promo + refunds wallet credits; a recipient-fault (`RECIPIENT_UNAVAILABLE`) doesn't. Four triggers: a LIVE drone **telemetry exception phase** (`FAILED`/`RETURNING`/`RETURNED`, still LIVE-only + drone-bound + never-auto-DELIVER), the **admin** `POST /admin/deliveries/:id/fail`, the **handoff-OTP lock** (5th wrong code auto-fails as `RECIPIENT_UNAVAILABLE`), and the sim stays happy-path-only (failed via admin). Every transition fans out a notification + WS status. Verified end-to-end with a mock publisher (no hardware).
17. ~~**i18n / localization** (the app/data lean Indonesia/Jakarta) and accessibility pass.~~ ✅ **Done** (server-side share). An in-house, **non-request-scoped** `I18nService.translate(key, locale, params?)` (en + id) — required because the **BullMQ worker** that fires delivery notifications has no `Accept-Language`, so locale is a persisted `User.locale` resolved by `userId` and translation is a pure `(key, locale, params)` function usable identically in the worker and request handlers. A `@Global` module + per-locale typed catalogs (`src/i18n/catalog/`); a no-throw fallback chain (locale → `en` → the key itself) so a missing key can never break a notification; a tiny `{placeholder}` interpolator (no ICU dep). **Localized:** the 5 delivery-stage + 8 exception notifications (title/body + the live-map drone-status label), the 2 transactional emails, and support (auto-ack + FAQs). Locale is **set** via `PATCH /users/me` (`locale`) and captured best-effort from `Accept-Language` at signup; the anonymous password-reset email derives locale from the header only (account-existence non-disclosure preserved); `GET /users/me` exposes `locale`. A catalog-completeness test gates drift. **Accessibility boundary** (honest): a headless API can't own UI a11y — its contribution is localized plain-language content + structured machine-readable payloads (the discrete `status` enum stays separate from the human `droneStatus` label, so a client maps the stable enum to its own accessible label) + a consistent error envelope + exposed `locale`. The actual a11y pass (screen-reader labels, focus order, contrast, semantic markup, dynamic type, reduced motion) lives in the **Expo mobile + admin-web repos**, not here. **Deferred (called out):** localizing the ~86 developer-facing error/validation literals (the global exception filter is the future entry point) and a real HTML email template engine (the mailer is still a dev-stub). **P3 complete.**

### How to choose
If the goal is **launch-readiness**: do P0 (auth recovery, payments, distance pricing, proof of delivery) — without these the app isn't shippable to paying users.
If the goal is **scale to 100k**: pair P0 with **ARCHITECTURE.md Phase 1** (worker tier + caching + pooling) — features and scale must advance together, since real payments and live tracking are exactly what fall over under load.
