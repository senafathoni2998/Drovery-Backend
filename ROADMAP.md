# Drovery ‚Äî Feature Recommendations

Prioritized advice on **what to build next**, grounded in what exists today. Ordered by
impact-to-effort. ‚úÖ = already added in the latest round of work.

## ‚úÖ Just shipped (this round)
- **Live drone tracking** ‚Äî backend simulates a moving drone; mobile polls every 4s and the marker **glides** between positions with a LIVE badge.
- **Status-change notifications** ‚Äî in-app + local banner on each change, and **remote Expo push** to registered devices.
- **Cancel delivery** UI wired to the existing endpoint.
- **Geocode-on-create** ‚Äî real pickup/dropoff coordinates ‚Üí a real flight path.
- **Support tickets persisted** + signed (HMAC + expiry) QR codes.
- **‚úÖ Distance-based pricing (P0)** ‚Äî `PricingService` now adds a haversine `distanceFee` ($1.50/km) from coords or geocoded addresses; `DeliveriesService` delegates to it so the quote and the stored price always agree. Mobile sends the addresses and shows a distance line.
- **‚úÖ Password reset (P0)** ‚Äî `POST /auth/forgot-password` + `/auth/reset-password` with hashed, single-use, 1-hour tokens (no email enumeration). Email send is behind `MailService` (logs the link in dev; swap in SendGrid/SES). Mobile has Forgot/Reset Password screens + a wired "Forgot password?" link, with deep-link token prefill.
- **‚úÖ Real Stripe payments (P0)** ‚Äî `StripeService` uses the live Stripe SDK when `STRIPE_SECRET_KEY` is set and a deterministic **mock** otherwise. Creating a delivery creates a **PaymentIntent** + a `Payment` row (idempotent); `POST /payments/webhook` verifies the signature and drives `Payment.status`.
- **‚úÖ Native Stripe card entry** ‚Äî backend mints a Customer + **SetupIntent + ephemeral key** (`POST /payment-methods/setup-intent`) and reconciles saved cards (`POST /payment-methods/sync`), real-or-mock. Mobile uses **`@stripe/stripe-react-native`'s PaymentSheet** (gated behind `EXPO_PUBLIC_STRIPE_PUBLISHABLE_KEY`; falls back to the manual form). *Requires a dev build + Stripe keys to exercise the native sheet.*
- **‚úÖ Proof of delivery (P0)** ‚Äî on `DELIVERED` the simulation auto-records a `ProofOfDelivery` (photo + final GPS + recipient + timestamp); `POST/GET /deliveries/:id/proof` lets the owner submit/view one. Image storage is behind `StorageService` (inline data URL / placeholder in dev; S3/Cloudinary when configured). Mobile shows a proof card on the delivery detail **and can capture a real photo** (expo-camera + current GPS ‚Üí upload, replacing the placeholder).
- **‚úÖ Security hardening** ‚Äî refresh-token rotation + revocation (hashed store, `jti`, reuse ‚Üí 401, logout endpoint), per-IP rate limiting (`@nestjs/throttler`, tighter on `/auth`), weak-JWT-secret boot guard in production, owner-scoped tracking lookups, and a CORS allowlist (`CORS_ORIGINS`). All verified live.
- **‚úÖ Observability** ‚Äî structured JSON logging (`nestjs-pino`) with per-request correlation ids (`X-Request-Id`, echoed back + propagated to logs) and auth-header redaction; **health probes**: `GET /health` (liveness) + `GET /health/ready` (checks DB + Redis, returns **503** when a dependency is down). Public, un-throttled, k8s-ready. **Sentry error tracking** (`@sentry/node`, DSN-gated no-op) reports unhandled 5xx from the global exception filter in both the API and worker. **Prometheus `GET /api/v1/metrics`** (`prom-client`): default Node metrics, an HTTP histogram labelled by route *template* (cardinality-safe), and a `drovery_queue_jobs{queue,state}` gauge from BullMQ ‚Äî the worker also serves it at `:9091/metrics`.
- **‚úÖ Horizontal scaling + containers** ‚Äî multi-instance-safe (Redis-backed rate limiting shared across replicas, bounded pg pool + PgBouncer pooling tier, role-split + cloud-ready Redis clients); multi-stage **Dockerfile** (one image runs api/worker/migrate) + **docker-compose** full stack (`--scale api=2 --scale worker=3`) + **GitHub Actions CI**.
- **‚úÖ Autoscaling milestone** ‚Äî **Kubernetes/Kustomize** manifests (`k8s/`, base + local/prod/loadtest overlays): api **HPA** on CPU + worker **KEDA** `ScaledObject` on BullMQ queue depth (Prometheus `max(waiting)+max(delayed)`), PDB, direct-to-Postgres migrate Job. **k6** load tests (`load/`, create‚Üítrack‚Üídeliver) + a `LOADTEST_BYPASS_THROTTLE` flag (non-prod) so a single-IP run measures real throughput. Validated with `kustomize build` + `kubeconform`; CI dry-runs on `kind`. *(Next: a real cluster run for actual scale-up numbers + Grafana dashboards.)*
- **‚úÖ Real-time tracking (push, horizontally scalable)** ‚Äî WebSocket tracking over raw `ws` (`WsAdapter`) backed by **Redis pub/sub**: the worker publishes each position/status change to `delivery:<id>:update`, and every API replica's subscriber fans it out to its local clients ‚Äî so a worker-computed update reaches a client on **any** instance (the in-memory gateway was dead in the worker-split topology). **Auth + ownership** on every subscription (JWT in the `?token=` handshake, `findOne` ownership re-check; tokenless ‚Üí `1008`). Polling (`GET /deliveries/:id`) is kept as an additive backstop ‚Äî **mobile still polls; WS migration is a separate mobile-repo task.** `drovery_ws_connections` gauge added. Verified cross-process. *(Next: dedicated realtime tier so sockets scale apart from the API.)*
- **‚úÖ Recipient handoff OTP (P1 #5)** ‚Äî the drone now stops at a new **`AWAITING_HANDOFF`** state on arrival (the sim never auto-delivers); finalizing as `DELIVERED` + recording proof requires the recipient's **6-digit code** via `POST /deliveries/:id/confirm-handoff`. Code is SHA-256-hashed (plaintext returned once on create, never re-exposed ‚Äî `handoffCodeHash`/`handoffAttempts` omitted from all reads), constant-time compared, single-use (atomic CAS), with an atomic **5-attempt lockout** (`423`). Owner-scoped; `401/409/400/404` on the error paths. Verified live + adversarially reviewed.
- **‚úÖ Serviceability gate (P1 #8)** ‚Äî "can we fly this?" checks: **service area** (point-in-circle, Greater Jakarta + Bandung), **no-fly zones** (route-vs-circle geometry, Jakarta airports), and **weather** (real-or-mock `WeatherService`, OpenWeather when keyed, **fail-open**). `POST /pricing/estimate` returns an advisory `serviceability` block; `POST /deliveries` rejects unflyable routes (**422** out-of-area / no-fly / unresolved-location, **503** weather hold) before any charge/queue. Reused `haversineKm` (moved to `common/geo-distance` to break a Pricing‚ÜîServiceability cycle). Verified live + adversarially reviewed.
- **‚úÖ Saved addresses / address book (P1 #7)** ‚Äî `/addresses` CRUD (jwt, owner-scoped, default-first, atomic single-default via `$transaction`, per-user cap) with **geocode-on-save** (best-effort; stores coords so future deliveries skip the rate-limited geocoder) + `GET /addresses/recent` derived (deduped) from delivery history. lat/lng bounded to WGS84. Verified live + adversarially reviewed.
- **‚úÖ Ratings & feedback (P1 #6)** ‚Äî once a delivery is `DELIVERED`, the owner can rate it 1‚Äì5 stars + optional comment via `POST /deliveries/:id/rating` (upsert, owner-scoped, `DELIVERED`-only ‚Üí `409`). Embedded in the delivery detail includes. Verified + adversarially reviewed.
- **‚úÖ Notification preferences (P1 #9, part 1)** ‚Äî `GET/PATCH /notifications/preferences` for per-user `pushEnabled` / `deliveryUpdates` / `promotions` / `quietHoursStart`/`End`. The in-app feed is **always** recorded; preferences gate only the **push** fan-out (`create()` takes a `category` and checks the matching toggle + quiet hours). Quiet hours are `[start,end)` wall-clock hours (wrap-around handled), evaluated in `NOTIFICATIONS_TZ` (**default `Asia/Jakarta`, not the UTC container**) and require both bounds together (`400` otherwise). Verified live + adversarially reviewed (the review caught the UTC-vs-WIB quiet-hours bug + the silent half-window).
- **‚úÖ Scheduled deliveries (P2 #10, part 1)** ‚Äî a future pickup window now **defers the whole lifecycle** instead of flying immediately. `POST /deliveries` derives the pickup UTC instant from `pickupDate + pickupTime` in the service tz (dependency-free `Intl` offset math), and a lead time > 60s (‚â§ 60-day horizon, else `400`) sets a new **`SCHEDULED`** status + `scheduledFor` and enqueues one delayed BullMQ **kickoff** job; at the window it **atomically flips `SCHEDULED ‚Üí PENDING`** (enqueue-before-flip so a transient retry recovers) and starts the normal simulation. Immediate/past pickups behave exactly as before; cancel works while scheduled (removes the kickoff job); new `?status=scheduled` filter (kept distinct from in-flight `current`). Designed via a 2-lens panel workflow; +20 tests (339 pass). Live-verified incl. the kickoff **firing at the pickup instant** (`SCHEDULED ‚Üí PENDING ‚Üí CONFIRMED`); live-verify caught a real bug (BullMQ `queue.add` rejects `:` in a custom jobId) and the review caught a retry-stranding ordering bug ‚Äî both fixed. *(Next: truly recurring deliveries.)*
- **‚úÖ Realtime support chat (P1 #9, part 2)** ‚Äî a SECOND WebSocket gateway at a distinct path **`/ws/support`** (the WsAdapter routes upgrades by exact pathname, so the mobile's tracking socket at `/` is untouched ‚Äî verified live). Mirrors the tracking tier: JWT-handshake auth (`1008` if tokenless), per-ticket **ownership** re-checked at subscribe AND send, dedicated-connection **Redis pub/sub** fanout (`support:ticket:<id>:messages`) so a message accepted on any replica reaches every subscribed client on every replica. Messages persist in a new **`SupportChatMessage`** thread (idempotent backfill seeds existing tickets); a **SYSTEM auto-acknowledgement** is seeded on ticket creation. **REST backstop**: `GET/POST /support/tickets/:id/messages` (history + send, identical effect to the WS frame). Built **human-agent-ready** (USER/AGENT/SYSTEM roles + tier-agnostic publisher) ‚Äî a future agent surface needs no schema change. `drovery_ws_support_connections` gauge. Designed via a 2-lens panel workflow; verified live (two clients realtime over the full publish‚ÜíRedis‚Üísubscribe loop, both gateways coexisting) + adversarially reviewed (caught a non-atomic seed write + a migration UUID portability gap, both fixed). *(Next: the agent/admin surface ‚Äî inbox, status transitions, optional AI-assist.)*

## P2 ‚Äî Growth & engagement

10. **Scheduled** ‚úÖ **/ recurring** *(recurring still open)* **deliveries.** A pickup far enough ahead now defers the whole lifecycle to that instant: `POST /deliveries` computes the pickup UTC instant from `pickupDate + pickupTime` in the service tz (`NOTIFICATIONS_TZ`), sets a new `SCHEDULED` status + `scheduledFor`, and enqueues a single delayed BullMQ **kickoff** job that (at the window) atomically flips `SCHEDULED ‚Üí PENDING` and starts the normal simulation. Cancelable while scheduled; `?status=scheduled` filter; bounded to a 60-day horizon. **Still open:** truly **recurring** deliveries (a schedule entity + a materialization worker) ‚Äî the one-off path is the foundation for it.
11. **Promo codes & wallet/credits** at checkout.
12. **Referral program** ("invite a friend, both get credit") ‚Äî cheap growth loop.
13. **Reorder / "send again"** from history; **favorites**.

## P3 ‚Äî Ops & platform maturity

14. **Operator/admin app or dashboard** ‚Äî today everything is the consumer side. Ops needs fleet status, exceptions, manual reassignment, refunds.
15. **Live drone telemetry ingestion** to replace the simulation (MQTT/drone gateway ‚Üí worker tier ‚Üí same tracking contract).
16. **Delivery exceptions** (failed drop, weather abort, return-to-base) as first-class statuses with user comms.
17. **i18n / localization** (the app/data lean Indonesia/Jakarta) and accessibility pass.

### How to choose
If the goal is **launch-readiness**: do P0 (auth recovery, payments, distance pricing, proof of delivery) ‚Äî without these the app isn't shippable to paying users.
If the goal is **scale to 100k**: pair P0 with **ARCHITECTURE.md Phase 1** (worker tier + caching + pooling) ‚Äî features and scale must advance together, since real payments and live tracking are exactly what fall over under load.
